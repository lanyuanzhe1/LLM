## 1. 项目背景与平台分析

### 1.1 讯飞平台总览

本次项目的核心平台是讯飞（科大讯飞）提供的两条主线：

| 平台 | 地址 | 定位 | 核心能力 |
|------|------|------|----------|
| **讯飞星辰 MaaS** | [training.xfyun.cn/modelSquare](https://training.xfyun.cn/modelSquare) | 模型精调与部署平台 | 零代码微调、LoRA/FFT、模型部署、GPU 算力 |
| **讯飞星辰 Agent** | [agent.xfyun.cn/home](https://agent.xfyun.cn/home) | 智能体构建平台 | 零代码建 Agent、知识库管理、工作流编排 |

两个平台**协同使用**构成完整的技术链路：
```
数据准备 → 向量化(Embedding API) → 知识库(Agent平台)
                                  → 模型微调(MaaS平台) → 模型部署 → Agent集成 → 上线发布
```

### 1.2 平台核心能力清单

#### 星辰 MaaS 平台
- **支持模型**：星火系列 (Spark Max/Lite/Mini/Pro) + 30+ 开源模型 (Qwen-2.5 系列、DeepSeek 系列等)
- **精调方式**：一句话精调（零代码）/ 专业精调（LoRA / FFT 全量微调）
- **数据格式**：Alpaca 格式、ShareGPT 格式，支持 JSON/JSONL/CSV/Parquet
- **辅助工具**：问答对抽取、数据增强、Prompt 工程（50+ 模板）
- **推理部署**：一键发布 API 服务、批量推理、多模型对比评估
- **免费资源**：新用户赠送 **10 小时 GPU 算力**

#### 星辰 Agent 平台
- **智能体类型**：指令型、工作流型、全自主开发型
- **知识库能力**：基础 RAG / Agentic RAG / Long RAG 三档策略
- **工作流节点**：大模型节点、知识库节点、代码节点、决策节点、工具节点等
- **发布渠道**：API、MCP Server、微信公众号、H5、星火APP
- **额外能力**：MCP 协议全链路支持（16,000+ MCP Server）、多模型 Prompt 对比

#### Embedding & Rerank 服务
- **文本向量化 API**：兼容 OpenAI SDK 格式，2560 维向量，支持 2K Token
- **双模式**：`para`（文档向量化）和 `query`（查询向量化）
- **Rerank 重排序 API**：精排候选文档，提升检索精度

> `★ Insight ─────────────────────────────────────`
> **关键认知**：讯飞提供的不是单一工具，而是一个**完整的垂直大模型构建生态系统**。MaaS 处理"训练"，Agent 处理"编排"，Embedding API 处理"向量化"，三者通过统一的 API 网关互通。理解这个三角架构是制定技术方案的基础。

## 2. 核心技术决策框架

### 2.1 微调 vs RAG：不是二选一，而是混合架构

这是整个项目最关键的架构决策。2025-2026 年行业共识非常明确：

| 决策维度 | 选 RAG | 选微调 | 混合架构（推荐✅） |
|----------|--------|--------|-------------------|
| 知识变更频率 | 频繁更新 | 静态知识 | 动态知识→RAG，静态→微调 |
| 行为控制 | 弱 | 强 | 语气/格式/拒答→微调 |
| 幻觉控制 | 有引用可追溯 | 权重内化 | 事实依赖→RAG |
| 更新成本 | 低（增删文档） | 高（重新训练） | 按需组合 |
| 延迟要求 | 50-200ms 额外 | 无额外延迟 | 依路由分流 |

**推荐方案：Pattern C — 混合架构（2026 生产级默认）**

                         │   用户输入问题    │

                  ┌─────────▼┐  ┌──▼───┐  ┌▼─────────┐
                  │ 事实查询  │  │知识问答│  │ 闲聊/其他 │

## 3. 阶段一：知识库准备与向量化

### 3.1 知识库清洗与预处理

这是整个项目的基础，**数据质量直接决定最终效果的上限**。

#### 3.1.1 文档格式统一

| 原始格式 | 处理方案 | 工具 |
|----------|----------|------|
| PDF（扫描版） | OCR 识别 → 文本提取 | 讯飞 OCR API |
| PDF（文字版） | 直接提取文本 | PyMuPDF / pdfplumber |
| Word (.docx) | 提取段落结构 | python-docx |
| 网页内容 | HTML → Markdown | 讯飞问答对抽取工具 |
| 纯文本 (.txt) | 格式检查，去噪 | 自写脚本 |
| 表格数据 | 转换为结构化文本描述 | pandas + 模板 |

#### 3.1.2 文本清洗规范

```
必做检查清单：
✅ 去除重复段落（基于语义相似度 > 0.95）
✅ 修正 OCR 识别错误（常见错别字词典替换）
✅ 统一术语表达（如：AI/人工智能/大模型 → 统一映射）
✅ 去除页眉页脚、水印等噪声
✅ 保留文档层级结构（章节、小节标题）
✅ 标记不可读内容（乱码、图片、公式）
```

#### 3.1.3 文档分块策略（Chunking）

分块可能是向量化中**最被低估但影响最大**的环节：

| 文档类型 | Chunk 大小 | 重叠量 | 策略 |
|----------|------------|--------|------|
| FAQ / 问答对 | 300-500 字符 | 50-100 | 每个 QA 独立分块 |
| 技术文档 | 500-800 字符 | 100-150 | 按标题层级切分 |
| 长报告/教材 | 800-1200 字符 | 200-300 | 按章节+语义边界 |
| 规范/标准文档 | 按条款切分 | 0-50 | 保留条款编号 |

**核心原则**：
- ✅ 优先采用**语义边界切分**，而非按字符数硬切
- ✅ 保留 10%-20% 的内容重叠，防止关键信息在边界处断裂
- ✅ 每个 Chunk 必须**语义自洽**（离开上下文也能独立理解）
- ✅ Markdown 文档使用**标题感知分块**（Header-aware chunking）

> `★ Insight ─────────────────────────────────────`
> **为什么 Chunking 这么重要？** RAG 系统的"检索"环节是在 chunk 级别进行的。如果 chunk 切得太碎（50字），语义信息丢失，检索效果差；切得太粗（2000字），噪声太多，LLM 找不到关键信息。最优实践是按**语义单元**切分——一个 chunk ≈ 一个完整的独立知识点。这往往比调 embedding 模型本身更重要。
> `─────────────────────────────────────────────────`

### 3.2 Embedding 向量化方案

#### 3.2.1 讯飞 Embedding API（推荐首选）

**关键参数**：
| 参数 | 值 |
|------|-----|
| API 地址 | `https://maas-api.cn-huabei-1.xf-yun.com/v1/embeddings` |
| 模型 ID | `sde0a5839` |
| 向量维度 | 2560 |
| 最大 Token | 2048 |
| 协议兼容 | OpenAI SDK 格式 |

**Python 调用示例**：
```python
import openai

client = openai.OpenAI(
    api_key="<YOUR_API_KEY>",
    base_url="https://maas-api.cn-huabei-1.xf-yun.com/v1"
)

def get_embedding(text: str, model: str = "sde0a5839") -> list[float]:
    """调用讯飞 Embedding API 获取文本向量"""
    response = client.embeddings.create(
        model=model,
        input=text,
    )
    return response.data[0].embedding
```

**双模式使用**（关键优化）：
```python
# 文档向量化 — 使用 para 模式
doc_embeddings = [get_embedding(chunk) for chunk in document_chunks]

# 查询向量化 — 使用 query 模式
query_embedding = get_embedding(user_question)
```

> ⚠️ **重要**：Embedding API 和星火大模型 API 的授权是**独立的**，需要在讯飞开放平台单独申请开通。

#### 3.2.2 备选方案：开源 Embedding 模型

| 模型 | 维度 | 优势 | 使用场景 |
|------|------|------|----------|
| BGE-M3 | 1024 | 多语言+多向量，中文突出 | 需要本地部署 / 自定义微调 |
| m3e-base | 768 | 中文优化，轻量 | 资源受限环境 |
| text-embedding-3-small | 768 | 高性价比 | 成本敏感 / 快速原型 |

#### 3.2.3 向量存储方案

| 阶段 | 推荐方案 | 适用规模 |
|------|----------|----------|
| **开发验证** | FAISS (Flat 索引) | < 10 万向量 |
| **测试上线** | FAISS (HNSW 索引) | 10 万 - 100 万向量 |
| **生产大规模** | Milvus / pgvector | 100 万+ 向量 |

**代码示例 — FAISS 向量库构建**：
```python
import faiss
import numpy as np

# 假设已通过 Embedding API 获得所有向量
vectors = np.array(all_embeddings).astype('float32')

# 归一化（余弦相似度）
faiss.normalize_L2(vectors)

# 创建 HNSW 索引
dimension = 2560  # 讯飞 Embedding 维度
index = faiss.IndexHNSWFlat(dimension, 32)  # 32 为连接数
index.add(vectors)

# 检索
def search(query_embedding, top_k=5):
    query = np.array([query_embedding]).astype('float32')
    faiss.normalize_L2(query)
    distances, indices = index.search(query, top_k)
    return distances[0], indices[0]
```

## 4. 阶段二：模型微调 (Fine-Tuning)

### 4.1 微调数据准备

#### 4.1.1 数据来源

基于已有知识库，构造微调数据集的主要途径：

| 途径 | 方法 | 产出效率 | 质量保障 |
|------|------|----------|----------|
| **问答对抽取** | 讯飞平台工具：上传 TXT/网页 → 自动切分 Q&A | 高（正确率 90%） | 需人工抽检 |
| **手动标注** | 领域专家编写 instruction-output 对 | 低但质量最高 | 专家审核 |
| **数据增强** | LLM 批量泛化种子数据（1→5倍） | 中等 | 人工筛选 |
| **历史数据迁移** | 从已有客服记录/论坛提取 | 依数据量而定 | 需清洗 |
| **Self-Instruct** | 用强模型生成训练数据 | 高 | 需质量过滤 |

#### 4.1.2 数据格式标准（Alpaca 格式）

```json
[
  {
    "instruction": "你是一个[领域]专家助手。请根据以下问题给出专业、准确的回答。",
    "input": "[用户的具体问题]",
    "output": "[期望的准确答案]",
    "system": "你是一位资深的[领域]专家，拥有[XX]年的从业经验。（可选）",
    "history": []
  }
]
```

**完整示例 — 假设是"建筑规范"垂类**：
```json
{
  "instruction": "你是一位建筑结构设计专家。请根据《建筑结构荷载规范》回答以下问题。回答需引用具体规范条款。",
  "input": "某办公楼位于8度抗震设防区，地上12层，高度45米。请问该建筑应属于哪一类抗震设防类别？",
  "output": "根据《建筑抗震设计规范》(GB 50011-2010)第3.1.1条规定：\n\n该办公楼属于**标准设防类（丙类）**。理由如下：\n\n1. 建筑高度45米<50米，未达到重点设防类（乙类）的高度阈值\n2. 12层办公楼属于一般性公共建筑\n3. 8度设防区本身不影响设防类别判定\n\n丙类建筑的地震作用应符合本地区抗震设防烈度的要求，抗震措施应符合本地区抗震设防烈度的要求。"
}
```

#### 4.1.3 数据量与配比

| 数据量 | 效果预估 | 建议 |
|--------|----------|------|
| < 500 条 | 效果不明显 | 先做数据增强 |
| 1,000 - 2,000 条 | **最小可行量** | 配合高质量样本 |
| 2,000 - 5,000 条 | 开始看到明显效果 | ⭐ 推荐初始目标 |
| 5,000 - 10,000 条 | 效果稳定 | 复杂领域推荐 |
| 10,000+ 条 | 接近上限收益递减 | 仅当资源充足 |

**数据集划分**：70% 训练集 / 15% 验证集 / 15% 测试集

#### 4.1.4 数据增强策略

利用讯飞平台或外部 LLM 对种子数据进行增强：

```
种子数据 (200-500条)
    │
    ├── 同义泛化：变换表达方式，保持语义不变
    ├── 同类泛化：保持逻辑结构，替换情境/对象
    ├── 增加约束：添加时间/顺序/范围等限制条件
    ├── 复杂场景：移植到更具体的真实场景
    ├── 考察方向变换：改变提问角度和侧重点
    └── 前提变换：改变基础条件或关键信息
    │
    ▼
增强后数据 (1,000-2,500条)
```

### 4.2 微调执行（讯飞星辰 MaaS 平台）

#### 4.2.1 平台操作流程

```
Step 1: 登录 → training.xfyun.cn/modelSquare
Step 2: 数据管理 → 创建数据集 → 上传 JSONL 文件
Step 3: 模型精调 → 创建模型 → 选择基础模型 + LoRA
Step 4: 配置参数 → 提交训练 → 监控 Loss 曲线
Step 5: 训练完成 → 发布为 API 服务
Step 6: 在线体验 / 批量推理 → 效果评估
```

#### 4.2.2 LoRA 微调参数推荐配置

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| **基础模型** | Qwen-2.5-7B-Instruct | 综合性价比最优 |
| **学习率** | 5e-5 | MQA 架构标准值 |
| **训练轮数 (Epochs)** | 4-5 | 2000-5000 条数据建议 4 轮 |
| **LoRA Rank (r)** | 16 | 7B 模型推荐值 |
| **LoRA Alpha** | 32 | 设为 Rank 的 2 倍 |
| **最大序列长度** | 2048 | 按最长样本调整 |
| **Dropout** | 0.05 | 结构化数据防过拟合 |
| **目标模块** | q_proj, v_proj | Qwen MQA 架构专用 |
| **数值精度** | bf16 | 节省显存，保持精度 |

> `★ Insight ─────────────────────────────────────`
> **LoRA 的核心直觉**：想象你要教一个已经精通英语的人学说一种方言。你不需要教他重新学语法和单词（那等于全量微调），只需要在他已有的知识上做一点点局部调整。LoRA 做的事就是这个——找到大模型中与你的任务最相关的几个"注意力头"（q_proj 和 v_proj），只在这些地方微调极少量参数（<1%）。这就是为什么 LoRA 只需要几分钟、一张显卡就能完成微调。
> `─────────────────────────────────────────────────`

#### 4.2.3 训练监控指标

训练过程中需重点关注：
- **Loss 曲线**：应持续下降并收敛，不收敛需调整学习率
- **验证集 Loss**：与训练 Loss 的差距应逐渐减小
- **过拟合信号**：训练 Loss 持续下降但验证 Loss 开始上升 → 减少 Epochs 或增加 Dropout

### 4.3 微调后评估

## 5. 阶段三：RAG 检索增强架构搭建

### 5.1 RAG 架构设计

#### 5.1.1 推荐架构：多路召回 + 重排序

```
用户问题
    │
    ├── 意图分析（关键词提取 + Query 改写）
    │
    ├── 三路并行召回 ────────────────────┐
    │   ├── 向量检索 (Embedding API)     │
    │   ├── 关键词检索 (BM25)            │
    │   └── 知识图谱检索（如适用）        │
    │                                     │
    └── 结果融合 → 粗排 Top 50-100 ───────┘
                    │
                    ▼
              Rerank 精排 Top 5-10
                    │
                    ▼
              上下文拼接 → LLM 生成 → 带引用输出
```

#### 5.1.2 各环节技术方案

| 环节 | 方案 | 技术要点 |
|------|------|----------|
| **向量检索** | 讯飞 Embedding API + FAISS | 2560 维 HNSW 索引 |
| **关键词检索** | BM25 (rank-bm25 库) | 精确匹配兜底 |
| **重排序** | 讯飞 Rerank API | BERT-based 精排模型 |
| **Query 改写** | LLM 改写原始问题 | 扩展搜索关键词 |

#### 5.1.3 RAG 代码架构示例

```python
class RAGPipeline:
    def __init__(self):
        self.embedding_client = EmbeddingClient()  # 讯飞 Embedding API
        self.bm25_index = BM25Index()              # 关键词索引
        self.reranker = RerankClient()             # 讯飞 Rerank API
        self.llm = LLMClient()                     # 微调模型
        self.vector_index = faiss.IndexHNSWFlat(2560, 32)

    def retrieve(self, query: str, top_k: int = 50) -> list[Document]:
        """多路召回"""
        # 向量检索
        query_embedding = self.embedding_client.embed_query(query)
        vector_results = self.vector_index.search(query_embedding, k=top_k)
        
        # 关键词检索
        keyword_results = self.bm25_index.search(query, k=top_k)
        
        # 融合去重 (Reciprocal Rank Fusion)
        fused = self.fuse_results(vector_results, keyword_results)
        return fused
    
    def rerank(self, query: str, candidates: list[Document], top_k: int = 5):
        """Rerank 精排"""
        return self.reranker.rerank(query, candidates, top_k=top_k)
    
    def generate(self, query: str, context: list[Document]) -> str:
        """基于检索上下文生成答案"""
        prompt = self.build_prompt(query, context)
        return self.llm.generate(prompt)
    
    def build_prompt(self, query: str, context: list[Document]) -> str:
        """构造带约束的生成提示词"""
        context_text = "\n\n---\n\n".join(
            f"[来源 {i+1}]\n{doc.text}" for i, doc in enumerate(context)
        )
        return f"""你是一个专业的领域助手。请**仅基于**以下提供的参考资料回答问题。

## 参考资料
{context_text}

## 回答要求
1. 如果资料中有答案，必须**引用具体来源编号**
2. 如果资料中没有答案，请明确说明"当前资料不足以回答此问题"
3. 保持专业、准确的表达

## 用户问题
{query}"""
```

### 5.2 性能优化要点

| 优化方向 | 技术手段 | 预期效果 |
|----------|----------|----------|
| 检索速度 | HNSW 索引替代 Flat | 从 O(N) → O(log N) |
| 存储压缩 | PQ 量化（2560→64维） | 存储减少 96% |
| 精度优化 | 混合检索 (向量+BM25) | 召回率提升 27-41% |
| 排序精度 | Rerank 模型引入 | Top3 准确率 68%→89% |
| 缓存策略 | 热点问题向量缓存 | 减少 70% 重复计算 |
